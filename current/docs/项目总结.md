大语言模型（Large Language Model, LLM）在自然语言处理领域取得了突破性进展\cite{openai2024gpt4,deepseekv3,qwen3}，其应用正在从通用场景向金融等专业领域快速渗透\cite{xie2024finben,chen2023discfinllm}。金融行业每天都在定时产生海量的标准化文档，如年报摘要、研究报告、行政处罚决定书、融资融券分析报告等，这些文档格式固定、对创造力要求低，但对准确性和规范性要求极高，这些模版化数据和重复性工作很容易被大语言模型所替代，进而大大提高工作效率。

【我们认为这些标准化的文档处理工作（具体是哪些，现在就可以说了）可以被模型所取代，这些任务本身不难、很容易完成、不要求创造力，主要是IF，所以大模型overqualified了，可以尝试让小模型替代（这是从任务的客观角度出发说为什么觉得小模型能胜任这件事，我们也可以看看通用IF上小模型的表现对比）】

【其次呢，从主观角度上出发，我们也很想让小模型去做这件事，为什么呢】
\begin{enumerate}
    \item 可以本地部署，不用闭源模型API，安全隐私性强，后面也可以放心内接企业知识库，因为任务其实不难，我们觉得其实值钱的是数据，国家也鼓励企业私有化部署
    \item 成本低
\end{enumerate}

【从上面两点的客观和主观角度出发，我们希望能微调出一个小模型帮助金融相关从业者和科研人员处理结构化文档，产出报告等等】
【那么做成这件事需要啥呢】

【首先我们需要一个可靠的benchmark，评估我们是否把这件事做好了做完了，其次需要指令微调数据做一个后训练SFT微调，但是我们通过文献调研，发现并没有完美符合需求的现有benchmark，于是我们选择自建benchmark，然后训练数据构造和bench数据构造采用同一套数据构建链路pipeline】

【具体来说，是将user prompt分部分构造：full_prompt=context+base_query+constraint组合，context来源于真实文本（具体说说有啥）、现有金融相关bench的开源数据文本、合成文本，base_query基于context和任务分类体系合成，任务分类体系是自建的两级分类体系（T1-T3），基本涵盖了重复性任务的各个场景，constraint组合是从自建constraint池中随机抽1-5条约束进行组合，然后通过GPT5审查将不合适的constraint组合改写，不合适分为两个方面，一个是constraint内部冲突，比如要求字数不超过300字和字数在400-600之间，一个是constraint和query描述冲突，这里再举一个具体例子，那么constraint池是怎么构建的呢，我们从IFEval、IFBench、CFBench、InfoBench、FIFE中广泛搜集了constraint，并结合我们的任务合成了一些新的constraint，仿照Infobench的constraint分类体系对每个constraint进行了打标，另外根据校验方式将其分为hard constraint和soft constraint】

【这样做完之后，我们就能得到prompt数据，与此同时我们设置完整的校验链路，对于hard constraint采用rule-based checker打分，对于soft constraint 采用LLM-as-a-judge的方式打分并给出reason，judge model选的ds-v4-flash，原因是它在IFbench表现不错，且成本较GPT低，均采用binary score，怎么算accuracy的可以给个计算公式，其实就是pass rate】

【有了数据，有了评测方法，我们就能构建起自己的benchmark，我们选了100条数据，覆盖各个任务场景和约束类型以及约束数量，对9个模型进行了评测，发现大小模型的差距其实并不大，且主要问题在于xx类型，认为可以通过针对性微调的方式实现comparable甚至反超的效果。这里做了一些数分】

【有了数据，我们就可以进行微调训练了，我们根据自建benchmark发现GPT5.4在这个领域表现断档的好，就让它作为teacher model，让它对2000多条训练数据生成回复，并进行打分，我们只要完全通过的，只要有1条约束没通过的就返回模型原回复、原始的full prompt和judge model判不过的原因，让其对原回复进行修复直到完全通过】
【最终拿着sft的输入数据我们转换为shareGPT的格式，在LLama Factory的训练框架下微调q wen-3-8B,最后在自建bench和通用bench上都实现了提升，验证了我们的猜想】

【因此总结在全过程中我们的贡献：】
\begin{enumerate}
    \item 建立了一个中文金融的指令遵循评测基准FinIF
    \item 建立了一套完整的自动化的易于scaling的数据构造链路，通过将context、query、constraint解耦，模块化操作，完全可以通过任务类型、约束数量、约束类型、约束组合、query提问方式等方式进行scaling
    \item 微调出了一个模型（起个名字），能直接部署，进行日常的金融文本信息抽取、报告撰写等工作
\end{enumerate}

【不足】
\begin{enumerate}
    \item 没有用agent，日常场景中直接上传一个完整Office文件情况比较多
    \item 没有做correctness评测维度，默认任务很简单，大小模型都能完成的很好
\end{enumerate}

【未来方向】
\begin{enumerate}
    \item 加上agent，一个是合成数据的，一个是生成回复和打分的
    \item 补上对correctness的实验
\end{enumerate}
